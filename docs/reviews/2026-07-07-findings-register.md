# DIMENSION SUMMARIES

## llm-instructions
The core RAG-extraction design is genuinely strong for a single-user tendering tool: per-field/per-category retrieval with [SOURCE|PAGE]-labeled context, anti-fabrication rules, instructor+Pydantic structured output, independent NLI citation verification with three-signal confidence, free-tier key rotation, and graceful no-key degradation. The two known defects have clear root causes in the prompt/verifier chain: (a) citation matching relies on exact string equality between the LLM-echoed filename and chunk.filename, with no verbatim-copy hard rule, no enumerated valid filenames, and no normalization — any drift (case, extension, Arabic Unicode forms) silently caps confidence at 0.3/low even when the quote is verbatim; (b) the submission_documents bucket is starved by the combination of the prompt's hard "skip other categories" rule (submission items are almost always commercial/eligibility in content) and cross-category semantic dedup that deletes the submission_documents copy in favor of earlier-extracted categories. Two further systemic gaps: the English-only NLI cross-encoder collapses the whole confidence pipeline to "low" on Arabic tenders (a core plan.md requirement), and no generation config is set anywhere (temperature defaults ~1.0, hurting verbatim-quote fidelity). The offer extraction/compliance prompts and schemas are markedly weaker than the summary/checklist ones (no field descriptions, no enums, no grounding rules). The default model id gemini-2.5-pro (a March-2025 model) is configurable but stale for mid-2026 and is the worst free-tier throughput choice for this short-context structured workload.
STRENGTHS:
- Retrieval-then-extract per field/category with [SOURCE:filename | PAGE:N]-labeled context (app/services/llm/context_builder.py) is exactly the right RAG shape for citation-grounded extraction — the LLM only ever sees retrieved excerpts, never asked to answer from memory.
- Independent NLI cross-encoder verification of every citation plus three-signal confidence (NLI 50% / retrieval 30% / LLM 20%) with requires_review flags (app/services/extraction/citation_verifier.py) is a legitimately sophisticated anti-hallucination defense that avoids self-verification bias — the right call for compliance-critical tender data.
- Strong anti-fabrication prompt hygiene in both main prompts: 'NEVER fabricate or infer', 'set value to null and confidence to 0.0 if not found', verbatim-quote requirement, and empty-list-if-none for checklists (app/services/llm/context_builder.py:79, :126-127).
- Mandatory-classification rule with explicit keyword mapping and a tender-safe default ('ambiguous -> mandatory', context_builder.py:124) encodes real domain judgment — missing a mandatory item risks disqualification.
- Instructor wrapper-model pattern (CategoryExtractionResponse instead of bare list, documented rationale in app/schemas/checklist.py:58-63) and thorough Field descriptions with ge/le/min_length constraints on the extraction/checklist schemas are well-shaped for Gemini structured output.
- Key rotation design (app/services/llm/gemini_service.py) — round-robin spread, rotate on quota/permission/leaked-key markers, fail-fast per-key validation retries so an exhausted key doesn't burn quota — is precisely fitted to the 8-key free-tier reality of this deployment.
- Graceful degradation is consistent at every LLM boundary: empty field/category on failure, LLMUnavailable -> 503 for offers, status lifecycle tracking (in_progress/completed/failed) — the works-without-LLM-key guarantee is actually honored.
- Context size is bounded by design everywhere (max 20 chunks x ~400 chars for extraction, 10k/file 30k/total chars for offers), so there is no context-overflow risk despite no explicit token counting.
- Multilingual embedding model (paraphrase-multilingual-mpnet-base-v2) with semantic-weighted hybrid search (alpha=0.7, documented as 'better for multilingual') means Arabic retrieval works even though queries are English.

## domain-fields
The domain modeling is fundamentally sound: the 13-field summary hits the core commercial envelope (bond, advance, retention, payment terms, contract type), the checklist's six buckets include an eligibility/pre-qualification category correctly flagged as bid/no-bid critical, SupplierOffer captures the full offer envelope the spec asks for, and the formula-preserving client-template writer is exactly the right deliverable mechanism. The three real gaps are: (1) several spec-listed, high-value summary fields are missing — performance bond %, liquidated damages/penalties, project duration (which the indirects engine needs), defects liability/warranty, and clarification deadline; (2) roughly half of rules.default.json is decorative — keywords.*, compliance.*, most naming.* formats, min_items/grouping_criteria, measurement.contract_type/quantity_tolerance, and default validity/payment terms are never read by any service, so the single user's main customization surface silently no-ops; (3) Arabic coverage is inconsistent — trade classification keywords are English-only (an Arabic BOQ classifies 0% of items), and retrieval queries for extraction/checklist are English-only. Offer comparison also ranks on raw total_price without VAT or currency normalization despite storing vat_included. None of these break the current English-tender happy path, but they are the difference between "works on the demo tender" and "right for an Arabic-market tendering department."
STRENGTHS:
- Evidence-first extraction design (Citation with document+page+verbatim quote, NLI verification, three-signal confidence, requires_review flag) on every summary field AND every checklist item — this is exactly the plan.md 'never fabricate, always cite' rule made structural, and it is the single most important domain-correctness property for tender work (app/schemas/extraction.py, app/schemas/checklist.py).
- The 13 summary fields prioritize the right commercial terms for a bid decision: tender bond, advance payment, retention %, payment terms, bid validity, and contract_type as a constrained enum (lump_sum/remeasured/unit_rate/cost_plus/design_build) matching plan.md's Lump Sum vs Remeasured requirement (app/services/extraction/field_definitions.py).
- Dedicated 'eligibility / pre-qualification' checklist bucket with prompt hint 'CRITICAL for bid/no-bid decisions' — correct domain emphasis that goes beyond plan.md's own category list (app/services/extraction/checklist_definitions.py:134-151).
- BOQItem.client_row_index + client_ref plus the formula-preserving template_writer (loads with formulas intact, writes only the rate column, two-pass rate-column detection so a generic 'Price'=total column can never be clobbered, rejects .xlsm to avoid silent macro loss) — precisely the right mechanism for populating a client's pricing template without breaking their formulas (app/models/boq.py:42-43, app/services/pricing/template_writer.py).
- SupplierOffer stores the full commercial envelope plan.md asks to extract from offers: total/currency/vat_included, validity_days+date, payment_terms, delivery_weeks+delivery_terms (incoterms), exclusions, deviations, missing_items, clarifications_needed, line_items — and OfferCommercialUpdate provides a complete manual-entry path preserving the works-without-LLM guarantee (app/models/supplier.py:98-166, app/schemas/offer.py:51-66).
- measurement.unit_mappings normalizes the actual messy unit aliases seen in regional BOQs (sqm/sq.m, cum/cu.m, nr/nos/ea, lm/lin.m, ls) — small, practical, market-neutral (config/rules.default.json:76-83).
- Scoring is configurable (weights + thresholds), and the semantics are domain-sane: missing price/delivery scores 0 rather than neutral (documented anti-gaming rationale), payment terms scored buyer-favorable by parsed net-days, supplier rating folded in at 10% (app/services/offer/scoring_service.py).
- Document versioning with addenda-supersede (content_hash dedup, is_superseded/superseded_by_id, 'addendum' deliberately first in classification precedence so 'Addendum 3 - revised specs' never classifies as specs) mirrors real tender document control (app/models/document.py, app/services/versioning/doc_classifier.py).
- Bilingual surface built into the data model, not bolted on: BOQItem.description_ar, Supplier.name_ar, Supplier.preferred_language driving per-supplier RFQ language, en/ar email templates, Arabic-aware chunking (app/models/boq.py:37, app/models/supplier.py:39,67, app/services/email/templates.py).
- HistoricalPrice is decoupled from any one project with explicit provenance (source, source_project_id, recorded_at) — matches plan.md's 'learn from historical sheets while keeping traceability' (app/models/historical.py).
- Indirects model (percentage-of-direct + duration-based roles + location factors, with zero-rate defaults so nothing is fabricated until the user configures rates) matches how estimators actually build indirects, and the additive dict-merge in rules loading lets the user extend trades/locations without replacing defaults (app/services/indirects/indirects_service.py, app/services/rules/rules_service.py:44-45).
- Deliverables bundle composition (Pricing_Summary with the direct-only vs direct+indirects grand-total distinction explicitly documented in the schemas, Pricing_Gaps by unpriced/needs-review/excluded, per-package comparison workbooks, packages register, briefs, manifest.json) matches plan.md's 'package all deliverables for submission' (app/services/deliverables/deliverables_service.py, app/schemas/pricing.py:45-51, app/schemas/indirects.py:41-47).

## architecture-structure
The root app/ tree is structurally healthy and appropriately lazy for a single-user tendering tool: one flat FastAPI package, 21 thin routers over feature-scoped service packages, plain SQLAlchemy models and Pydantic schemas, no repository/DI/hexagonal layers anywhere, and the largest module is only 443 lines. Nothing in app/ references the legacy bidops-ai/ tree. The genuine problems are residue, not architecture: a dead auth/multi-tenancy model cluster (User/Organization/AuditLog) inherited from the salvage tree that still creates three tables and is kept green by tests, a services-import-API inversion where two services reach into app.api.search for a private singleton, ~80 lines of copy-pasted LLM/search stack wiring across three routers, an alembic setup that is never executed by any deployment path while startup uses create_all, and untracked clutter at the repo root (package-lock.json with no node project, JVM crash logs, verification artifacts, the whole bidops-ai/ tree). All findings are deletions or small moves — no new abstraction is needed, and no finding rises to 'broken'.
STRENGTHS:
- Flat, honest layout with zero speculative layering: routers call service classes directly (e.g. OfferService() constructed inline in app/api/offers.py), services take db: AsyncSession as a plain argument, no DI container, no repositories, no interfaces-for-one-implementation. This is exactly the laziest structure that works.
- Pure-ASGI middleware (app/middleware.py) instead of BaseHTTPMiddleware, chosen specifically so the SSE progress stream and file downloads are never buffered, with the required ordering documented at the add_middleware call sites in app/main.py — minimal and deliberate.
- In-memory dict progress store (app/services/progress.py, 107 lines) explicitly rejecting Redis for single-user SSE — the right call, stated in the docstring.
- Heavy-model laziness is consistent and correct: module-level Docling converter cache (app/services/parsing/pdf_parser.py), lazy ChromaDB client/embedding function (app/services/indexing/embedding_service.py), off-by-default warmup flag, and model construction moved off the event loop (app/services/document_service.py _get_embedding_service_async) — protects tests, no-LLM-key mode, and HF Space cold starts.
- Uniform parser contract: all 8 parsers return the same ParsedDocument dataclass and degrade to warnings instead of raising (app/services/parsing/base.py), so messy tender uploads can never crash the ingest pipeline.
- Rules system is a 62-line two-file JSON deep-merge (app/services/rules/rules_service.py) with typed defaults in app/schemas/rules.py — a configurable business-rules engine done without a database table or admin framework.
- Error shape is consistent and documented: 4xx via FastAPI's default {detail} everywhere, with a single deliberate {error:{type,message,request_id}} envelope reserved for unhandled 500s and rate-limit 429s (app/errors.py docstring explains the split).
- Export formats are coherent for the domain: everything tabular is openpyxl .xlsx (summary report, offer comparison, supplier DB, deliverables workbooks), pricing round-trip preserves client formulas via a dedicated template_writer, and PDF goes through one Jinja template + print CSS (app/templates/reports/pdf_report.html + app/static/css/pdf_report.css).
- Templates are lean and well-split: 36-line base.html, v1 project page vs v2 workbench with 8 small _partial includes (app/templates/workbench/), one 883-line shared stylesheet, no frontend build step — appropriate for a server-rendered single-user UI.
- The startup lifespan in app/main.py documents its one real invariant (snapshot restore MUST precede first engine connection) instead of hiding it behind an abstraction.

## security-deployment
The engineering hygiene here is genuinely strong for a single-user tool: git history is verifiably clean of secrets (git log -S "AIzaSy" across all branches is empty), .gitignore and .dockerignore both exclude .env so no key can reach GitHub, the HF Space repo, or the Docker image; the backup dataset is created private=True; restore uses tarfile's data filter with staging and integrity checks; and the container runs non-root with secrets delivered via Space Secrets. The dominant problem is not code but a configuration contradiction: the Space is deployed PUBLIC while the README (line 55) and the code's own comments assume it is private — meaning every tender document, BOQ price, supplier contact list, and delete/upload/LLM endpoint is world-reachable, and a stranger's destructive action gets snapshotted over good state within ~60 seconds. Secondary gaps are all small-effort: the main document upload has no size cap (unlike the offers upload, which does), rate limiting ships off and would mis-key behind HF's proxy anyway, and the zip parser caps members/depth but not decompressed bytes. Fixing the visibility toggle alone removes most of the real risk; the rest is cheap tightening that matters only while the Space stays public.
STRENGTHS:
- Secret handling is verifiably clean: .env is untracked, .gitignore covers it at every level (root and bidops-ai/), .dockerignore excludes .env/.env.*/data/.git from the image build context, and a full-history search (git log --all -S "AIzaSy") confirms no API key was ever committed to any branch, including the HF Space remote.
