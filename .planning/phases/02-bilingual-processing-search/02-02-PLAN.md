<objective>
Create the document chunking and vector indexing pipeline that transforms parsed documents into searchable embeddings stored in ChromaDB.

Purpose: After documents are parsed (Phase 1), they need to be split into appropriately-sized chunks and embedded into vector space for semantic search. This plan builds the indexing backbone that the search layer (Plan 02-03) will query.

Output: ChunkingService that splits documents at semantic boundaries with Arabic-aware separators, EmbeddingService that manages per-project ChromaDB collections with multilingual embeddings, and integration into the existing document_service.py to auto-index after parsing.
</objective>

   - Docstring: "Document indexing services for chunking and vector embedding."

2. **Create `app/services/indexing/chunking_service.py`**:
   - Define `DocumentChunk` dataclass with fields:
     - `chunk_id: str` -- format: `"{document_id}_p{page_number}_c{chunk_index}"` for text, `"..._t{index}"` for tables
     - `document_id: int` -- FK to Document table
     - `page_number: int` -- source page number (for citations)
     - `text: str` -- original chunk text (for display)
     - `text_normalized: str` -- normalized text (for embedding and BM25 indexing)
     - `language: str` -- "ar", "en", "mixed", or "unknown"
     - `chunk_type: str` -- "text" or "table"
     - `section_name: str | None` -- detected section heading if any
     - `char_start: int` -- character offset in page text
     - `char_end: int` -- character offset end
     - `metadata: dict` -- additional metadata (filename, etc.)
   - Implement `ChunkingService` class:
     - Constructor params: `max_chunk_chars: int = 400`, `overlap_chars: int = 50`
     - Arabic-aware separators list: `["\n\n", "\n", ". ", ".\u060C ", "\u060C ", ", ", " "]` (includes Arabic comma \u060C)
     - `chunk_document(document_id: int, pages: list[PageContent], filename: str) -> list[DocumentChunk]`:
       - For each page: first emit table chunks (each table as ONE chunk, never split), then split page text at semantic boundaries
       - Import and use `normalize_for_search` from `app.services.text_processing` for `text_normalized`
       - Import and use `detect_language` from `app.services.text_processing` for language detection
       - Use recursive splitting: try first separator, if chunk still too large try next separator
       - Apply overlap: when splitting, include `overlap_chars` characters from end of previous chunk at start of next
       - Track char_start/char_end offsets relative to page text
     - `_split_text(text: str) -> list[tuple[str, int, int]]`: recursive character splitter returning (text, start_offset, end_offset) tuples
     - `_table_to_text(table: dict) -> str`: convert table dict (headers + data) to pipe-separated text representation for embedding. Format: "| header1 | header2 |\n| val1 | val2 |"
     - `_detect_section_heading(text: str) -> str | None`: simple heuristic -- if chunk starts with a short line (< 80 chars) followed by a newline, treat it as a section heading

3. **Create `app/services/indexing/embedding_service.py`**:
   - Implement `EmbeddingService` class:
     - Constructor takes `persist_dir: str` (from config) and `model_name: str` (from config)
     - Lazy-initialize `chromadb.PersistentClient(path=persist_dir)` on first use (avoid startup cost)
     - Lazy-initialize `SentenceTransformerEmbeddingFunction(model_name=model_name, device="cpu", normalize_embeddings=True)` on first use (~420MB model download on first run)
     - `get_collection(project_id: int) -> chromadb.Collection`: uses `client.get_or_create_collection(name=f"project_{project_id}", embedding_function=self._ef, metadata={"hnsw:space": "cosine"})`. CRITICAL: Always pass embedding_function to avoid dimension mismatch (Pitfall 3 from research).
     - `index_chunks(project_id: int, chunks: list[DocumentChunk]) -> int`: Add chunks to collection. Use `collection.upsert()` (NOT `add()`) for idempotency. Embed the `text_normalized` field. Store all metadata (document_id, page_number, language, chunk_type, section_name, char_start, char_end, filename). Return count of chunks indexed.
     - `delete_document_chunks(project_id: int, document_id: int) -> None`: Delete all chunks for a specific document using `collection.delete(where={"document_id": document_id})`. Call this BEFORE re-indexing to avoid orphan chunks (Pitfall 7 from research).
     - `delete_collection(project_id: int) -> None`: Delete entire project collection (for project deletion).
     - `get_collection_count(project_id: int) -> int`: Return number of chunks in collection.

4. **Update `app/config.py`**:

     - `chunk_max_chars: int = 400` -- max chunk size in characters (tuned for 128-token limit with Arabic)

1. **Modify `app/services/document_service.py`**:
   - Add imports at top:
     ```python
     from app.config import get_settings
     from app.services.indexing.chunking_service import ChunkingService
     from app.services.indexing.embedding_service import EmbeddingService
     ```
   - Create module-level lazy-initialized services (same pattern as pdf_parser's lazy converter):
     ```python
     _chunking_service = None
     _embedding_service = None

     def _get_chunking_service():
         global _chunking_service
         if _chunking_service is None:
             settings = get_settings()
             _chunking_service = ChunkingService(
                 max_chunk_chars=settings.chunk_max_chars,
                 overlap_chars=settings.chunk_overlap_chars,
             )
         return _chunking_service

     def _get_embedding_service():
         global _embedding_service
         if _embedding_service is None:
             settings = get_settings()
             _embedding_service = EmbeddingService(
                 persist_dir=settings.chroma_persist_dir,
                 model_name=settings.embedding_model,
             )
         return _embedding_service
     ```
   - In `process_documents_batch()`, AFTER the successful parse block (after `doc.status = DocumentStatus.COMPLETED.value` and storing extracted_text/tables_json/metadata_json), ADD the indexing step:
     ```python
     # Phase 2: Chunk and index document for search
     try:
         chunking_svc = _get_chunking_service()
         embedding_svc = _get_embedding_service()

         # Delete any existing chunks for this document (re-upload case)
         embedding_svc.delete_document_chunks(project_id, doc_id)

         # Chunk the parsed document
         chunks = chunking_svc.chunk_document(
             document_id=doc_id,
             pages=parsed.pages,
             filename=filename,
         )

         # Index chunks into ChromaDB (CPU-bound embedding, run in thread)
         if chunks:
             chunk_count = await asyncio.to_thread(
                 embedding_svc.index_chunks, project_id, chunks
             )
             logger.info(
                 "Indexed %d chunks for %s (doc_id=%d)",
                 chunk_count, filename, doc_id,
             )

         # Enrich metadata with chunk info
         import json as json_module
         existing_meta = json_module.loads(doc.metadata_json) if doc.metadata_json else {}
         existing_meta["chunk_count"] = len(chunks)
         existing_meta["languages_detected"] = list(set(c.language for c in chunks))
         doc.metadata_json = json_module.dumps(existing_meta, ensure_ascii=False, default=str)
     except Exception as idx_exc:
         # Indexing failure should NOT fail the document parse
         logger.warning(
             "Indexing failed for %s (doc_id=%d): %s. Document parsing succeeded.",
             filename, doc_id, idx_exc,
         )
     ```
   - Add `import asyncio` to the imports if not already present (it should be since SSE uses it, but verify).
   - CRITICAL: The indexing step is wrapped in try/except so indexing failures don't fail the document parsing. The document is still marked COMPLETED even if indexing fails. This is intentional -- search is a secondary capability, not a gate on document ingestion.
   - CRITICAL: Use `asyncio.to_thread()` for `embedding_svc.index_chunks()` because embedding computation is CPU-bound (sentence-transformers inference).
